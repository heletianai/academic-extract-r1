## #008 TRL 要求 reward callable 带 `__name__`

**现象**：GRPO 首次冒烟 90 秒内崩：`AttributeError: 'RewardLogger' object has no attribute '__name__'`（grpo_trainer.py:301）。

**根因**：TRL 0.22 注册 reward 函数日志名时直接取 `reward_funcs[i].__name__`——普通函数有这个属性，实现了 `__call__` 的类实例没有。本地 153 项单测全绿却没抓到，因为单测从不实例化真 GRPOTrainer（需要 GPU）。

**修复**：`RewardLogger.__init__` 加一行 `self.__name__ = "reward_v1"`，二次冒烟通过。

**可迁移教训**：trainer 集成层的隐式 API 契约（框架对你传入对象的属性假设）单测覆盖不到，只能真环境冒烟抓。「上卡先跑 90 秒冒烟关卡」的纪律在本项目第 3 次兑现价值。

**30 秒口述**：我的 reward 是个带状态的类实例（要在线记录字段熵、gate 分布做监控），TRL 内部拿 `__name__` 当日志名，函数有类没有，上卡 90 秒就崩了。一行修复，但教训是集成层契约必须真环境冒烟——我所有训练脚本从此都带一个 90 秒关卡。

## #009 组内去重率尾部俯冲（高置信 prompt 采样收敛）

**现象**：temp 1.0 冒烟 5 步，gate 全 0、reward 冲 0.99+，一切看起来很好——但组内去重率逐 call 下滑：0.688 → 0.281。

**怀疑链**：先排除权重漂移（lr 5e-6 × 5 步动不了多少权重）→ 锁定 prompt 难度分层：抽取任务答案唯一，SFT 后模型对简单样本高置信，temp 1.0 下采 8 条全收敛到同一个正确 JSON → 全对全同 → 组内 advantage = 0，这类组在正式 run 中纯烧算力。

**判别点**：这是「塌向正确」而非 reward hacking（reward 高 + gate 0 佐证），处置是升温不是改 reward。

**修复**：temp 1.2 对照冒烟——去重率 0.938→0.781（均值 0.86 vs t1.0 的 0.55），gate 仍全 0。正式 run 采 t1.2。

**可迁移教训**：GRPO 的梯度信号来自组内差异。对答案唯一的确定性任务，采样温度是 advantage 信号存活的第一杠杆；监控里必须有一条「组内多样性」线（我用组内去重率），它归零比 loss 异常更早预警训练失效。

**30 秒口述**：GRPO 靠组内相对差异算 advantage，抽取任务答案唯一，模型置信后 8 条采样全一样，advantage 直接归零。我在冒烟期从去重率曲线抓到这个趋势，升温到 1.2 解决。这也是我后来能秒判 #013 行为塌缩的原因——同一条监控线。

## #010 GRPO 500 步权重全程空转

**现象**：500 步 GRPO 跑完，评测 overall 0.901——与 SFT 基线在每个字段、每个置信区间上**完全一致**。

**怀疑链→实锤**（这条链是本项目最重要的方法论资产）：
1. 数字太一致 → 第一怀疑不是「训练没效果」，而是「**同一个模型被测了两次**」；
2. 逐条 diff 200 条预测 → 200/200 逐字相同，坐实怀疑;
3. 校验最终 LoRA 与全部 checkpoint（step 100–500）的 adapter 权重 md5 → **全部等于 SFT 起点的逐位拷贝**——optimizer 从未写入参数（任何非零更新必然改变浮点位）；
4. 但训练日志 KL 全程 0.8–1.35 波动，「看起来在训练」——查 TRL PEFT 模式源码：ref model = disable adapter 后的 base，所以 KL 量的是「SFT 相对 base 的固定距离」，与训练进程无关。**KL 非零 ≠ 在训练**。

**根因**：`unsloth from_pretrained(LoRA 目录)` 加载出的 adapter 是推理态（冻结），非官方 notebook 的「基座 + get_peft_model」轨道。灌入 SFT 权重（set_peft_model_state_dict）后 md5 仍不变——说明还有第二层，见 #011。

**修复**：见 #011（原生 TRL + PEFT 重实现）。

**可迁移教训**：
1. **RL 冒烟必检「权重在动」**：N 步后 adapter md5 ≠ 起点，自此成为我所有 RL 训练的第五条冒烟标准；
2. 评测数字与基线完全一致 → 第一优先怀疑同一模型被测两次，而不是解释成「训练无效」；
3. KL 非零 ≠ 在训练（PEFT ref=base 时 KL 起点即非零）；loss 有值只证明前向可算。

**30 秒口述**：GRPO 跑完和 SFT 每个 CI 都一样，我第一反应不是「白训了」而是「是不是同一个模型测了两遍」——diff 发现 200 条预测逐字相同，再查全部 checkpoint 的 md5 全等于 SFT，坐实权重从没更新过。最迷惑的是 KL 曲线全程非零，看起来像在训练——其实 PEFT 模式下 ref 是 base，量的是固定距离。从此我的 RL 冒烟标准多了一条：跑 N 步查 adapter md5 必须变。

## #011 unsloth 慢路训练态失守 + vLLM 版本判死 → 原生 TRL 定案

**现象**：#010 的加载侧修复（正确轨道 + 显式灌权重）后，权重**仍然**空转，且 reward 序列与冒烟逐字重放。

**排查两线**：
- **unsloth 线**：其 GRPO 支持围绕 `fast_inference=True`（vLLM colocate）官方场景设计，`for_inference/for_training` 态切换装饰器（rl.py L203–444）在 transformers 慢路 rollout 下训练态恢复失守（机制定位到行号，修复优先于逐行考古）；
- **vLLM 线**：设计期 pin 的 vllm==0.6.6 装成且 torch 未动，但报 `Qwen3ForCausalLM not supported`——0.6.6 是 2024.12 的版本，早于 Qwen3 发布，pin 本身就是错的；升 0.8.x 需要连升 torch 全链（此前实测过的版本地狱），30 分钟止损闸内不可行，判死。

**终解**：原生 TRL + PEFT 完全绕开 unsloth：`AutoModel + PeftModel.from_pretrained(..., is_trainable=True) + enable_input_require_grads + gradient_checkpointing`，**绝不 import unsloth**（它会全局 patch TRL）。冒烟第五标准通过（md5 4039f4df → 88cfa5af），且 16.7s/it **反而快于** unsloth 慢路的 18.2s/it——它的优化在没走通的路径上本来就无效。

**可迁移教训**：
1. PEFT 加载默认推理态，`is_trainable=True` 必须显式（与 #010 同族坑，两处独立踩中）；
2. 加速库的收益只存在于其设计路径上；偏离官方场景（vLLM colocate → transformers 慢路）时，它的 patch 层可能反而是故障源；
3. 依赖 pin 要带环境条件：pin 一个早于目标模型发布的推理引擎版本，是设计期没做「版本 × 模型支持矩阵」核对的直接后果。

**30 秒口述**：修完加载还在空转，我做了个关键决策：不再修 unsloth，直接原生 TRL+PEFT 重写训练入口，因为 unsloth 的 GRPO patch 是为 vLLM colocate 设计的，我在慢路上等于用户量为零的路径。重写后不仅权重真的动了，速度还反超——它的优化在坏路径上本来就没生效。

## #012 多轮 prompt 从截断输出抄 schema 漏字段

**现象**：多轮冒烟机制指标全活（search_rate 0.5 / answered 0.85 / 轮数正常 / md5 变化正常），但 gate 率 100%——32 条全部 schema 不合法。

**怀疑链→实锤**：GPU 上取样肉眼看 answer 完全合法 → 本地零 GPU 复现 reward 判定 → errors 明细钉死：`limitation_mentioned: missing`。gold 是 8 字段，多轮 system prompt 里的 schema 说明只写了 7 个——它是从 `head -c 400` 的截断输出抄写的，第 8 个字段恰好在截断点之外。模型忠实地按 prompt 输出了 7 字段，被 validator 全判非法。

**修复**：MT prompt 的 schema 段改为**逐字复用训练数据内的 SFT system 原文**（与权重内化的口径恒一致，数据变了自动跟着变），多轮协议说明做后缀追加。

**可迁移教训**：关键配置（schema / 接口 / 协议文本）禁止从截断输出、摘要或记忆重建，必须程序化引用原文。截断丢的那一个字段，代价是整轮冒烟 100% 判废——而且这类错误机制层指标完全正常，只有业务层明细能钉死。

**30 秒口述**：多轮冒烟 gate 率 100%，但输出肉眼看全合法——最后在 reward 的 errors 明细里钉死：prompt 里的 schema 少写了一个字段，因为它是从截断的终端输出里抄的。修复方式不是补那个字段，而是让 prompt 程序化复用数据里的 SFT system 原文——从根上消灭「两份 schema 各自维护」这个错误类别。

## #013 多轮 RL 行为塌缩（search avoidance）

**现象**：多轮正式 run（penalty：零检索 −0.2 / 不作答 −0.5）warmup 结束后 **6 步内**：search_rate 0.5→0.03、answered_rate→1.0、组内 reward std 1.0→0.03、熵 0.20→0.06、reward 锁死 0.65 附近——全员收敛到「零检索直答」稳态。

**归因（期望值算账）**：直答 = F1(0.85–0.9) − 0.2 ≈ **0.65，无下行风险**；检索轨迹 = 若未收针则 gate −1.0 − 0.5 = **背着 −1.5 的下行风险**。组内对比下直答 advantage 恒正，GRPO 高效地淘汰了检索行为——正是 Search-R1++ 论文里 answer avoidance 的镜像（search avoidance）。**action penalty 是天平不是开关：两侧罚值之差就是你给行为标的价**。

**止损**：方差归零 + 熵塌 = 梯度信号已死，剩余 232 步（约 ¥12 / 5.5h）是纯空烧 → step ~25 杀进程止损，学费 ¥1.2。塌缩曲线与 28 个 call 的明细完整存档（`runs/grpomt-full-0722-collapsed-013.log`）。
诚实注脚：杀之前最后 3 个 call search_rate 回升到 0.22–0.41，疑似 β·KL 把策略往 SFT 先验拉回的对抗效应——持续性未观察即被截断，不作为结论。

**修复（数值消融，框架不动）**：penalty 0.2/0.5 → **0.8/0.3**（直答封顶 F1 − 0.8 ≈ 0.15 必亏；检索下行风险 −1.5 → −1.3）。扩展冒烟 20 步（5 步冒烟对行为塌缩是盲区——拐点在 step 7+，四轮冒烟全没看到它）→ 双线存活且同升 → 重挂 250 步全程健康收敛：**检索率 100%、作答率 100%、平均检索恰 1.00 次**——penalty 定价的精确收敛。同模型同数据，只改两个 reward 数字，行为从「永不检索」翻转为「全员恰一搜」（对照曲线见 `docs/figures/fig2-collapse-vs-fix.png`）。

**可迁移教训**：
1. 行为型 reward 的冒烟窗口必须覆盖 lr 满血后 ≥10 步——塌缩发生在 warmup 结束的瞬间，之前不会露头；
2. action penalty 两侧的罚值差 = 对行为的定价，失衡就塌向无风险臂；先对各行为臂算期望值再定数值；
3. RL 空转有两型：**权重空转**（#010，md5 抓）与**行为空转**（本条，权重在动但方差归零 + 熵塌 = 学习信号已死）——第五标准抓不到后者，组内 std / 熵 / 行为率三条曲线是唯一预警。

**30 秒口述**：正式 run 挂上 6 步就塌了：模型学会永不检索。我给两个行为臂算期望值——直答 0.65 无风险，检索背着 −1.5 的风险，组内对比直答恒赢，这是 Search-R1++ answer avoidance 的镜像。方差归零说明梯度已死，我立刻止损杀掉省下 ¥12。修复只改了两个数字：把直答罚到必亏、把检索的下行风险调轻——重训后检索率 100%、平均检索恰好 1.0 次，行为精确收敛到定价点。这次事故让我真正理解「reward 工程就是行为工程」。

## 横向沉淀：这六条共同说明什么

1. **监控线是资产**：md5（权重在动）、组内去重率（advantage 活着）、std/熵/行为率（信号没死）、gate 明细（错在哪个字段）——每条事故都是被某条监控线抓获的，没有一条是「跑完才发现」。
2. **止损纪律**：#010 跑满 500 步才发现（学费 ¥5.9），#013 在 step 25 就杀（学费 ¥1.2）——差距就是监控线判读能力的进步。
3. **每条事故都进了下一次的冒烟标准**：四标准 → 五标准（+md5）→ 六标准（+loss mask 断言）→ 行为窗口扩展（≥lr 满血后 10 步）。事故不复发的机制是标准化，不是记性。
